什么是 Sub-Agents？

专业化分工

就像一个高效的开发团队，每个Sub-Agent都有自己的专业领域。有的专攻前端，有的精通后端，还有的专门负责代码审查。再也不用一个人包打天下了！

上下文隔离

每个Agent都有自己的"工作空间"，不会被其他任务的细节干扰。主对话保持整洁，专注于高层决策，就像CEO只需要看报告摘要一样。

可复用资产

一次创建，处处使用。优秀的Sub-Agent可以在团队间共享，甚至开源给社区。将最佳实践固化为可执行的智能工具！

Sub-Agent配置详解

掌握配置文件的存储位置和格式规范，是使用Sub-Agents的基础 📁

文件存储位置

Sub-agents以Markdown文件形式存储，支持两种存储位置，遵循明确的优先级规则：

项目级代理

.claude/agents/
  • 作用域：仅在当前项目中可用
  • 优先级：最高
  • 版本控制：可随项目代码一起管理
  • 适用场景：项目特定的专业代理

用户级代理

~/.claude/agents/
  • 作用域：所有项目中都可用
  • 优先级：低于项目级代理
  • 版本控制：个人全局配置
  • 适用场景：通用的常用代理

优先级规则：当同名代理在两个位置都存在时，项目级代理(.claude/agents/)会覆盖用户级代理(~/.claude/agents/)

文件格式规范

每个Sub-agent都定义在一个Markdown文件中，采用标准的YAML frontmatter + 内容主体的结构：

your-sub-agent-name.md
---
name: your-sub-agent-name
description: Description of when this subagent should be invoked
tools: tool1, tool2, tool3 # Optional - inherits all tools if omitted
---

Your subagent's system prompt goes here. This can be multiple paragraphs
and should clearly define the subagent's role, capabilities, and approach
to solving problems.

Include specific instructions, best practices, and any constraints
the subagent should follow.
字段 必需 说明
name 是 唯一标识符，使用小写字母和连字符
description 是 自然语言描述，AI根据此字段进行自动委派
tools 否 逗号分隔的工具列表，省略则继承所有工具
model 否 指定模型(haiku/sonnet/opus)，用于性能优化
color 否 终端显示颜色，便于区分不同代理输出

最佳实践：description字段是自动委派的关键，建议使用"MUST BE USED"或"PROACTIVELY"等强提示词来提高触发准确性

Sub-Agents解决了什么核心问题？

从单一AI助手到专家团队，Sub-Agents革命性地解决了AI编程中的五大痛点 🎯

🧠 上下文污染与认知过载

问题：传统AI助手在长时间对话中，上下文窗口会被大量无关信息填满，导致"认知颠簸"和"指令漂移"，AI开始忘记核心任务或产生混乱的输出。

解决：每个Sub-Agent拥有独立的"干净"上下文窗口，专注于单一任务，避免信息污染，确保输出质量和准确性。

🎓 缺乏专业化深度

问题：通用AI助手试图"包打天下"，但在特定领域（如安全审查、性能优化、数据库设计）难以达到专家级水平，输出质量参差不齐。

解决：为每个专业领域创建高度专注的AI专家，通过精心设计的系统提示和工具配置，实现真正的"术业有专攻"。

🛡️ 安全性与权限控制缺失

问题：单一AI助手拥有所有工具权限，存在安全风险。一个代码审查任务不应该有删除文件或执行危险命令的能力。

解决：Sub-Agents实现精细化权限控制，每个Agent只能访问完成其任务所必需的最小工具集，遵循"最小权限原则"。

📈 可扩展性瓶颈

问题：单一上下文窗口有物理限制，复杂任务会快速消耗token，导致对话"失忆"或需要频繁重启会话，严重影响工作连续性。

解决：通过上下文隔离，实现"感知上的无限上下文"。主对话保持轻量，复杂任务在独立空间处理，理论上支持无限长的开发会话。

🔄 团队标准化与知识复用困难

问题：个人的提示工程技巧难以标准化和共享，团队成员的AI使用水平参差不齐，无法保证一致的代码质量和开发流程。

解决：Sub-Agents配置文件可版本控制、团队共享，将最佳实践固化为可执行的数字资产，实现"AI专家团队"的标准化部署。

核心洞察

Sub-Agents不只是功能升级，而是编程思维的范式转变。 它将"单打独斗"的AI助手转变为"专业协作"的AI团队， 从根本上解决了复杂软件开发中的上下文管理、 专业化深度和团队协作三大核心挑战。

完整项目开发提示词

复制此提示词，一键启动Sub-Agents团队开发
# 📋 项目需求
开发一个简单的web投票系统

# 🎯 功能要求
这个应用需要实现以下功能：
1. 显示投票问题和选项
2. 接受用户的投票
3. 展示每个选项的当前票数

# 🛠️ 技术栈
后端：Python + Flask应用
前端：HTML + CSS + JavaScript
存储：不需要数据库，内存存储

# 🚀 开发流程
1. 请 api-designer 智能体为我设计一个简单的Web投票应用的REST API端点。
   要求：请以清晰的Markdown格式提供API规格说明,API规格说明文档要求详细，包括地址和出入参的详细规定，保证前后端agent能够根据文档开发出前后端代码。

2. 使用上一步生成的API规格说明,请 backend-developer 智能体为我编写一个完整的Flask应用，文件名为 app.py。

3. 使用第一步生成的API规格说明,请 frontend-developer 智能体创建与Flask后端交互的前端文件。
   要求：创建 templates/index.html, static/style.css, 和 static/script.js。

4. 使用第一步生成的API规格说明，请主智能体完成前后端测试和修复工作，保证前后端能够正常的调用工作。

完整执行流程

输入提示词 → Agent自动识别 → 专家响应 → 高质量输出

只需要按顺序输入这三个提示词，就能从零开始构建一个完整的Web投票应用！每个Agent都会专注于自己的领域，确保输出的专业性和质量。

需要注意的挑战与限制 ⚠️

技术无完美，了解局限性才能更好地驾驭Sub-Agents 🎯

🧠 "白板问题"

核心问题：每个Sub-Agent启动时都是"失忆"状态，无法访问主对话的历史记录和上下文信息。

影响：需要开发者手动传递上下文，或通过文件方式进行信息共享，增加了使用复杂度。

⏰ 性能与延迟

核心问题：Sub-Agents启动时需要重新收集上下文，可能增加任务执行的延迟时间。

影响：复杂工作流可能消耗大量Token，执行时间较长，需要平衡效率和成本。

⚙️ 配置学习曲线

核心问题：设计高质量的Sub-Agent需要深入理解提示工程、权限控制和工作流编排。

影响：初学者可能需要大量实践才能创建出真正有效的专业化Agent。

🐛 调试复杂性

核心问题：多Agent工作流出错时，很难确定是哪个环节出了问题，调试变得复杂。

影响：需要更多的日志记录和中间检查点来定位问题所在。

🎲 自动委派的不确定性

核心问题：AI自动选择Agent时可能出现判断错误，选择了不合适的专家。

影响：可能需要手动干预或重新设计Agent的description来提高准确性。

⚠️ 系统已知Bug

核心问题：Claude Code存在"名称覆盖"问题，内置推断系统可能覆盖自定义的系统提示。

解决方案：避免使用过于描述性的Agent名称，如用"code-checker-alpha"替代"code-reviewer"。

克服限制的最佳策略

文件化沟通

使用PLAN.md等文件在Agent间传递上下文，避免"白板问题"

逐步构建

从简单Agent开始，逐步建立复杂工作流，降低学习成本

显式控制

关键任务使用显式Agent调用，避免自动委派的不确定性
